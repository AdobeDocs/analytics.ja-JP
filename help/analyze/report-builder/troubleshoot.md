# Report Builder のトラブルシューティングとベストプラクティス

Report Builder の配信を最適化する方法や、時々発生する可能性のあるエラーメッセージのリストを示します。

## Report Builder 5.0 のユーザーと 5.1 ワークブックの表示 {#section_C29898775999453FABB5FB0E098415C8}

ディメンションと分類の区切り文字をアンダースコア文字(_)から||. この変更は、Report Builder 5.0のユーザーが分類リクエストを使用してReport Builder v5.1ワークブックを開く場合の互換性に影響します。 v5.1より前のバージョンのワークブックを開くたびに、シリアル化されたすべての分類リクエストがこの形式に変換されます。

これは、次のような前方互換性の問題を引き起こします。v5.1に変換した後、Report Builder v5.0でワークブックを共有した場合、そのユーザーは分類リクエストを認識できなくなります（実際には、「_」は検索しますが、v5.1のシリアル化された「||」は認識されません）。

ARB v5.1 ワークブックを分類のリクエストで開くときには、次のような不具合が発生します。

* ワークブックを開くと、次の警告が表示されます。「このワークブックは、Report Builder バージョン 5.1 を使って最後に保存されています。このバージョンには、このコンピューターにインストールされているバージョンの Report Builder と互換性のない機能が含まれています。このワークブックを更新する前に、Report Builder の最新バージョンにアップグレードすることを強くお勧めします。」
* 分類付きのARBリクエストを右クリックしても、Report Builderのコンテキストメニュー（リクエストの編集、依存リクエストの追加など）は表示されません。
* 「すべて更新」を実行する場合、3つ目のボタンをクリックするか、リクエストマネージャーフォームからリクエストのセットを更新すると、分類リクエストはエラーなく実行されます。 ただし、分類の値は書き込まれません。
* 引き続き、リクエストマネージャーを開き、適切なリクエストに到達するまで1行ずつ移動することで、リクエストを編集できます。
* リクエストを編集し、すべてのパラメーターを同じままにして「完了」をクリックすると、応答が適切に書き込まれます。 実際、リクエストを編集すると、応答レイアウトパラメーターが再シリアライズされるので、問題が解決されます。 時間はかかりますが、回避策があります。

## Report Builder の認証の問題 {#section_FD79104DF1414FE2B36591606C963DE6}

Report Builder では、レポートスイートからデータリクエストを作成する場合、認証が必要になります。[!DNL Analytics] の設定やネットワークの状況によっては、Report Builder へのログインで問題が発生することがあります。

**無効なログイン会社名**

このエラーは、通常、ログイン会社が誤って入力された場合、またはネットワークアクティビティの問題が発生した場合に発生します。 次の手順を実行します。

* ログイン会社のスペルをチェックして、入力ミスや不要なスペースがないことを確認します。
* 同じログイン会社名を使用して Analytics にログインし、ログイン会社名が正しいことを確認する。これらの認証情報を使用してログインできない場合は、組織の管理者に正しいログイン会社名について問い合わせてください。

**ファイアウォール**

Report Builderでは、ポート80と443を使用します。 これらのポートが組織のファイアウォールを通じて許可されていることを確認します。 その他のファイアウォールの除外については、アドビの内部IPアドレスも参照してください。

## リクエストの最適化に関する推奨事項 {#section_33EF919255BF46CD97105D8ACB43573F}

次の要因によって、リクエストがより複雑になり処理が遅くなる場合があります。

**配信速度を低下させる可能性のある要因**

* 数時間以内に、過多のブックマーク、ダッシュボードおよびReport Builderワークブックがスケジュールされていた問題を修正しました。
* ほぼ同じ時間に、過多のReport Builderワークブックがスケジュールされていた問題を修正しました。 この場合、レポート API キューはバックログされます。

**ワークブックの実行時間を長くさせる要因**

* 分類の大幅な増加。
* リクエストの日付範囲の長期化。

   **例**：現在の年の設定を使用し、日の精度で分類したトレンドリクエストを作成するとします。年度の終わりにリクエストを行うと、年度の初めに作成したリクエストよりも多くの期間が返されます

   `(January 1 - January 30 versus January 1 - December 31.)`

**ワークブックの配信エラーを引き起こす原因**

ワークブック内の複雑な Excel 式（特に日付と時間を含むもの）。

**0 を返す（値を返さない）セル**

Excelのシート名にアポストロフィまたは一重引用符があると、report builderは値を返しません。 （これはMicrosoft Excelの制限です）。

**個別のリクエストのパフォーマンス**

処理速度は、次の設定の影響を受ける場合があります。

| 設定 | パフォーマンスの向上 | パフォーマンス低下 |
|--- |--- |--- |
| 分類と分類の順序 | 少数 | 多 |
|  | 例：A を Z で分類する場合、A の項目数は常に Z の項目数より少なくする必要があります。逆の場合はリクエスト時間が大幅に増加します。 |
| 日付範囲 | 狭い範囲 | 広範囲 |
| フィルタリング | 特定のフィルタリング | 最頻使用フィルタリング |
| 精度 | 集計 | 1 時間ごと<ul><li>毎日</li><li>毎週</li><li>毎月</li><li>四半期ごと</li><li>毎年</li></ul> |
| エントリ数 | 小さいデータセット | 大きなデータセット |


**スケジュール時間**

24 時間にわたり、スケジュールを分散させます（以下の表を参照）。既存のブックマーク、ダッシュボードおよびReport Builderワークブックを共に近い時間にスケジュール設定すると、遅延が発生する場合があります。

より大規模で複雑なリクエストを早朝にスケジュールし、業務日中に手動での取り込みと更新を可能にします。

| スケジュール時間 | 午前1時午前2時。 | 午前2時午前7時。 | 午前7時 — 午後6時。 | 午後6時 — 真夜中 |
|--- |--- |--- |--- |--- |
| Report Builderの使用 | 静か | 非常に多い | クライアント側の使用。<br>多くのユーザーが、ローカルからの更新や「すぐに送信」のリクエストを行います。<br>追加事項として、スケジュールされたワークブックがタイムアウトしたときに API キューがクリアされるかを検証します。 | ビジーでない |

**タイムアウト**

予定レポートは4時間後にタイムアウトします。 システムがさらに3回スケジュールを試み、失敗する可能性があります。 （一般に、データセットが大きいほど、実行時間が長くなります）。予定レポートは [!DNL Analytics] レポートおよび Report Builder の次の場所で表示できます。

* [!DNL Analytics]: **[!UICONTROL Favorites]** > **[!UICONTROL Scheduled Reports]**

* Report Builder: Click **[!UICONTROL Management]** in the [!UICONTROL Add-ins] tab in Excel.

## エラーメッセージの説明 {#section_3DF3A1EEDAD149CB941BEABEF948A4A5}

Report Builder の使用中に表示されることのあるエラーメッセージのリストです。

>[!NOTE] ここに紹介するエラーメッセージは一部のみであり、すべてを網羅したものではありません。エラーの解決法について詳しくは、管理者にお問い合わせください。

**この機能は開いているワークブックでのみ適用できます。**

Excelでワークブック(スプレッドシートドキュメント)が開いていない状態で、report builderツールバーのアイコンの1つをクリックすると、このメッセージが表示されます。 また、スプレッドシートを開くまでツールバーは無効になります。 ただし、このエラーを発生させずに、ツールバーが有効な状態でオンラインヘルプアイコンをクリックすることはできます。

**をアクティブ化する前に、を終了[!UICONTROL Request Wizard]する必要がありま[!UICONTROL Request Manager]す。**

とは機能的に [!UICONTROL Request Manager] リンクさ [!UICONTROL Request Wizard] れていますが、で行われた操作を完了またはキャンセルする前に、での作業を [!UICONTROL Request Manager] 開始することはできません [!UICONTROL Request Wizard]。

**この範囲に関連付けられているリクエストはありません。**

このエラーメッセージは、スプレッドシートのセルにリク [!UICONTROL From Sheet] エストが含まれていな [!UICONTROL Request Manager] い場合に、のボタンをクリックすると表示されます。

スプレッドシート内のどのセルにリクエストが含まれているかを識別するには、の表に表示されている個々のリクエストをクリックしま [!UICONTROL Request Manager]す。 リクエストがセルに関連付けられている場合、テーブルでリクエストが選択されると、セルがハイライト表示されます。

**選択されている範囲が無効です。**&#x200B;別の範囲を選択してください。

スプレッドシートのセルが選択され、既にリクエストがマッピングされている場合、このエラーが発生します。 セルにマッピングされたリクエストを削除するか、別のセル範囲を選択してマッピングします。

セルを削除する場合は、リクエストを含むセルを検索し、リクエストを削除してから、セルを削除する（行または列を削除する）ことが重要です。

**この機能を使用する前に Excel のセルの編集を終了してください。**

Excelのセルが編集モー *ドで* 、Report Builderのアイコンの1つをクリックすると、このエラーメッセージが表示されます。 Excelのセルの編集モードは、セルが選択され、カーソルがセル内に表示されることを意味します。 また、Excelの上部にあるバーに直接入力すると、Excelのセ [!UICONTROL Formula] ルで編集モードにな [!UICONTROL Name Box] ります。

**選択した範囲は別のリクエストの範囲と交差します。**&#x200B;選択を変更してください。

セルのセットを既にスプレッドシートにマッピングしている場合は、このエラーが表示されます。

新しいリクエストを追加する前に、どのセルがマッピングされているかを判断する1つの方法は、を閉じて開 [!UICONTROL Request Wizard] くことです [!UICONTROL Request Manager]。 次に、リクエストの概要の表に表示される項目を1つずつ選択します。 リクエストを選択するたびに、リスト内のリクエストマッピングを含む対応するセルがハイライト表示されます。

これは、複数のセルを複数の領域にマッピングする前に、セルにハイライト、行や列の情報、書式のスタイルを付けることを検討する必要がある理由の1つです。
